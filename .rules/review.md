# Submission Review Workflow

사용자가 `.ipynb`, `.py`, `.txt`, `.md` 등의 학습 답안을 리뷰해 달라고 요청할 때 적용한다.

## 리뷰 전 확인

1. 사용자가 지정한 모든 파일을 읽는다.
2. notebook은 코드 셀, 실행 순서, 출력, 오류와 하드코딩된 경로를 함께 확인한다.
3. 글 답안은 코드 결과와 일치하는지, 개념을 자기 말로 설명했는지 확인한다.
4. 현재 Exercise의 `코드 과제 C 문항`, `글 과제 T 문항`과 각각의 최소 통과 기준을 직전 세션 맥락에서 확인한다.
5. 코드 파일은 C 문항에만, 글 파일은 T 문항에만 대응시켜 체크리스트를 만든다. 코드 결과를 해석하는 T 문항은 두 파일을 함께 대조한다.
6. 과제 문항이 충분히 구체적이었는지 확인한다. 문항이 일반적이거나 모호했다면, 그 모호함 때문에 나온 일반적인 답변을 핵심 오답으로 몰지 않는다.
7. 제시 당시 C/T 문항이나 최소 통과 기준이 없었다면 제출물을 학습 목표에 맞춰 최선으로 평가하되, 리뷰 단계에서 새로운 필수 문항을 만들어 미통과시키지 않는다.

## 제출물별 채점 표

리뷰 결과는 다음 순서로 구분한다.

### 코드 과제

- 각 `C` 문항: `충족`, `보완 필요`, `해당 없음`
- 실행 오류, 출력, 시간 정렬, feature/target, split, leakage와 평가 행을 코드 최소 통과 기준에 따라 확인

### 글 과제

- 각 `T` 문항: `충족`, `보완 필요`, `해당 없음`
- 코드 출력과의 일치, 질문에서 요구한 개념과 지정된 구체성만 확인

### 추가 개선 제안

- 사전에 요구하지 않았지만 학습에 도움이 되는 내용은 `추가 개선 제안`으로 분리한다.
- 추가 개선 제안은 핵심 오류가 아니며 통과를 막는 조건으로 사용하지 않는다.
- 다음 Exercise에서 필수로 확인하려면 그때 새 C/T 문항과 최소 통과 기준으로 사전에 제시한다.

## 필수 평가 항목

1. Correctness: 코드와 설명이 사실 및 실행 결과와 일치하는가
2. Modeling Logic: 예측 문제와 코드 흐름이 논리적으로 연결되는가
3. Data Leakage Risk: 미래 정보나 전체 데이터 정보가 학습 과정에 섞이는가
4. Evaluation Quality: 분할 방식, 기준 모델, 지표와 해석이 적절한가
5. Possible Improvements: 현재 학습 수준에서 가장 가치 있는 개선은 무엇인가

아직 모델이나 평가가 없는 Exercise에서는 해당 항목을 억지로 채점하지 말고 `현재 실습 범위 밖`이라고 명시한다.

## 피드백 단계

- 첫 리뷰에서는 가장 중요한 문제부터 Hint를 제공한다.
- 사용자가 막히거나 더 구체적인 도움을 요청하면 Guidance로 진행한다.
- 다시 시도한 뒤에도 해결되지 않거나 일부 구현을 요청하면 Partial Solution을 제공한다.
- Full Solution은 사용자가 명시적으로 요청한 경우에만 제공한다.

각 단계에서 이미 해결한 문제를 반복해서 붙잡지 않는다.

리뷰에서 문제를 지적할 때는 단순히 "문제가 있다"고 끝내지 않는다. 특히 결측치, 중복 행, 시간 정렬, Data Leakage처럼 초급자가 판단 기준을 모를 수 있는 항목은 다음을 함께 설명한다.

1. 왜 문제인지
2. 지금 Exercise의 범위에서 반드시 처리해야 하는지, 아니면 다음 단계로 미뤄도 되는지
3. 처리한다면 어떤 순서로 처리해야 하는지
4. 잘못 처리하면 어떤 모델링 오류가 생기는지

다만 Full Solution에 해당하는 완성 코드는 사용자가 명시적으로 요청한 경우에만 제공한다.

## 통과 판단

- 핵심 학습 목표를 자기 말이나 코드로 설명할 수 있는지 본다.
- 사소한 오탈자나 다듬어지지 않은 문장만으로 통과를 지연하지 않는다.
- 과제 질문이 요구한 구체성보다 더 구체적인 답을 리뷰에서 사후 요구하지 않는다. 더 구체적인 답이 필요했다면 보완 질문 또는 다음 Concept Check로 제시한다.
- C 문항에서만 요구한 내용을 글 답안 누락으로 처리하거나, T 문항에서만 요구한 설명을 코드 누락으로 처리하지 않는다.
- 선택형 또는 `제출 불필요` 성찰 질문은 통과 조건으로 사용하지 않는다.
- 사전에 없던 baseline 선택 이유, 일반화 한계, 인과관계, publication lag, revision, 지표 해석 등을 리뷰 중 새 필수 조건으로 추가하지 않는다.
- 핵심 오개념, 실행 불가능한 코드, 누수 위험, 잘못된 평가 방식은 반드시 수정하게 한다.
- 통과 시 잘한 점, 수정된 점, 아직 남은 약점, 다음 학습 내용을 구체적으로 알린다.
- 통과 처리 후 `.rules/tracking.md`를 적용한다.

## 수정 권한

리뷰 요청만 받은 경우 사용자 답안 파일을 직접 수정하지 않는다. 사용자가 명시적으로 수정을 요청한 경우에만 답안 파일을 변경한다.
