# Part I. 데이터 표 자체를 정확히 읽기

## 1. 행, 관측치, 중복 행, 중복 날짜, 반복 값

### 1.1 반복했던 혼동

초기에는 같은 숫자가 반복되거나 같은 컬럼이 보이는 것을 중복 행과 혼동했다.

하지만 다음 세 가지는 완전히 다르다.

| 구분 | 뜻 | 예 |
|---|---|---|
| 반복 값 | 서로 다른 관측 시점에 같은 값이 나옴 | 1월 CPI=100, 2월 CPI=100 |
| 중복 날짜 | 날짜가 같지만 다른 컬럼 값은 다를 수 있음 | 6월 값이 두 행 |
| 완전 중복 행 | 모든 컬럼 값이 같은 행이 반복됨 | 날짜와 모든 값이 동일 |

### 1.2 작은 예시

| date | index | status |
|---|---:|---|
| 2024-01-01 | 100 | first |
| 2024-02-01 | 100 | first |
| 2024-02-01 | 101 | revised |
| 2024-02-01 | 101 | revised |

- 1월과 첫 번째 2월 행의 `100`은 반복 값이다.
- 2월 행 세 개는 중복 날짜다.
- 마지막 두 행은 완전 중복 행이다.

### 1.3 코드

```python
import pandas as pd

df = pd.DataFrame({
    "date": ["2024-01-01", "2024-02-01", "2024-02-01", "2024-02-01"],
    "index": [100, 100, 101, 101],
    "status": ["first", "first", "revised", "revised"],
})

df["date"] = pd.to_datetime(df["date"])

print(df.duplicated())
print(df.duplicated().sum())

print(df.duplicated(subset=["date"], keep=False))
print(df.duplicated(subset=["date"], keep=False).sum())
```

### 1.4 API 정확히 이해하기

#### `DataFrame.duplicated(subset=None, keep="first")`

- 필요한 이유: 중복 기준에 맞는 행을 찾는다.
- `subset=None`: 모든 컬럼이 같은지를 본다.
- `subset=["date"]`: 날짜만 같은지를 본다.
- `keep="first"`: 첫 번째 행은 `False`, 이후 중복은 `True`.
- `keep=False`: 중복 그룹의 모든 행을 `True`로 표시.
- 반환값: 행별 `True/False`를 가진 새 `Series`.
- 원본 변경: 하지 않는다.
- 재할당: 단순 확인에는 불필요하다.

#### `DataFrame.drop_duplicates(...)`

- 반환값: 중복이 제거된 새 `DataFrame`.
- 원본 변경: 기본값 `inplace=False`에서는 하지 않는다.
- 재할당: 결과를 계속 쓰려면 필요하다.

```python
cleaned = df.drop_duplicates()
```

다음 코드는 실행되어도 `df`가 바뀌지 않는다.

```python
df.drop_duplicates()
```

이 실수는 실제로 반복됐다. “메서드를 호출했다”와 “이후 데이터가 바뀌었다”는 같은 뜻이 아니다.

### 1.5 왜 시계열 모델링에서 중요한가

`shift(-1)`은 “다음 달”을 이해하지 않는다. 단지 현재 정렬에서 **다음 행**을 가져온다.

월별 데이터에 같은 월이 두 행이면 다음과 같은 잘못된 정렬이 생길 수 있다.

| 현재 행 | 다음 행 | `shift(-1)` 결과 |
|---|---|---|
| 2024-02 첫 행 | 2024-02 중복 행 | 같은 달 값 |

모델은 이것을 다음 달 target으로 오해하게 된다.

## 2. 날짜 변환, 날짜 정렬, 월별 연속성은 서로 다르다

### 2.1 반복했던 혼동

다음 세 작업을 여러 차례 같은 것으로 생각했다.

1. 날짜 변환: 문자열을 datetime으로 바꾸기
2. 날짜 정렬: 과거부터 미래 순서로 행 배치하기
3. 날짜 연속성: 중간 월이 빠지지 않았는지 확인하기

### 2.2 반례

```text
2024-01-01
2024-02-01
2024-04-01
```

이 날짜는:

- datetime일 수 있다.
- 오름차순일 수 있다.
- 하지만 2024년 3월이 빠져 있어 월별 연속적이지 않다.

`is_monotonic_increasing == True`는 “중간 월이 없다”는 증거가 아니다.

### 2.3 코드

```python
import pandas as pd

df = pd.DataFrame({
    "date": ["2024-04-01", "2024-01-01", "2024-02-01"],
    "value": [104, 100, 102],
})

# 1. 날짜 변환
df["date"] = pd.to_datetime(df["date"])

# 2. 날짜 정렬
df = df.sort_values("date").reset_index(drop=True)

print(df["date"].is_monotonic_increasing)  # True

# 3. 월별 연속성
expected = pd.date_range(
    start=df["date"].min(),
    end=df["date"].max(),
    freq="MS",
)
actual = pd.DatetimeIndex(df["date"])
missing_months = expected.difference(actual)

print(missing_months)  # 2024-03-01
```

### 2.4 월초와 월말 빈도

- `freq="MS"`: Month Start, 매월 첫날
- `freq="ME"`: Month End, 매월 말일

데이터가 월말 날짜인데 `MS`로 예상 날짜를 만들면 모든 날짜가 불일치할 수 있다. 빈도는 데이터의 실제 날짜 규칙에 맞춰야 한다.

### 2.5 체크 순서

월별 데이터에서는 다음을 각각 확인한다.

```python
df["date"] = pd.to_datetime(df["date"])
df = df.sort_values("date").reset_index(drop=True)

print("정렬:", df["date"].is_monotonic_increasing)
print("중복 날짜:", df["date"].duplicated().sum())
print("값 결측:", df["value"].isna().sum())

expected = pd.date_range(df["date"].min(), df["date"].max(), freq="MS")
missing = expected.difference(pd.DatetimeIndex(df["date"]))
print("누락 월:", missing)
```

## 3. 값의 결측과 행의 부재

### 3.1 두 표의 차이

#### A: 2월 행은 있지만 값이 없음

| date | value |
|---|---:|
| 2024-01 | 100 |
| 2024-02 | NaN |
| 2024-03 | 103 |

#### B: 2월 행 자체가 없음

| date | value |
|---|---:|
| 2024-01 | 100 |
| 2024-03 | 103 |

A는 값 결측이고 B는 누락 월이다.

이 차이는 `shift()`와 `rolling()`에서 매우 중요하다.

```python
# B에서 행 기반 shift(1)를 하면
# 2024-03의 이전 행은 2024-01이 된다.
df["lag1"] = df["value"].shift(1)
```

코드는 “직전 행”을 가져왔지만 우리는 이를 “직전 달”로 잘못 해석할 수 있다.

### 3.2 핵심

> 행 기반 lag가 달력 기준 lag가 되려면 날짜 빈도가 연속적이어야 한다.

# Part II. 한 학습 행의 시간 구조

## 4. Feature month, prediction time, target month, lag source month

### 4.1 가장 중요한 수작업 표

1개월 뒤 산업생산지수를 예측한다고 하자.

| feature month | prediction time | lag1 source | current feature | target month | target |
|---|---|---|---:|---|---:|
| 2024-02 | 2024-02 말 | 2024-01 | 108.4 | 2024-03 | 108.9 |

각 역할은 다음과 같다.

- Feature month: 입력 정보가 배치된 행의 기준월
- Prediction time: 실제 예측을 수행한다고 가정하는 순간
- Lag source month: lag feature의 원래 값이 나온 과거 월
- Target month: 맞혀야 할 미래 월
- Forecast horizon: prediction time에서 target month까지의 거리

### 4.2 한 행에 미래 target이 있어도 왜 괜찮은가

학습표는 모델이 연습할 문제와 정답을 함께 담는다.

```text
현재까지 아는 정보 X  →  나중에 확인된 정답 y
```

과거 학습 행에 미래 target을 정답으로 붙이는 것은 정상이다. 문제는 그 미래 target을 **입력 X에 넣는 것**이다.

```python
# 정상: y로 사용
y = model_df["target_next_month"]

# 누수: X에 사용
X = model_df[["lag1", "target_next_month"]]
```

### 4.3 `shift()`는 날짜가 아니라 행을 이동한다

```python
df["target_next"] = df["value"].shift(-1)
df["lag1"] = df["value"].shift(1)
```

정렬된 월별 연속 데이터에서:

- `shift(-1)`: 다음 행 값을 현재 행으로 가져온다.
- `shift(1)`: 이전 행 값을 현재 행으로 가져온다.

| date | value | `shift(1)` | `shift(-1)` |
|---|---:|---:|---:|
| 2024-01 | 10 | NaN | 20 |
| 2024-02 | 20 | 10 | 30 |
| 2024-03 | 30 | 20 | NaN |

2024-02 행에서:

- `shift(1)` 결과 10은 1월 값
- `shift(-1)` 결과 30은 3월 값

### 4.4 API

#### `Series.shift(periods=1)`

- 목적: 행 위치를 기준으로 값을 이동한다.
- `periods=1`: 이전 행 값을 현재 행에 배치.
- `periods=-1`: 다음 행 값을 현재 행에 배치.
- 반환값: 같은 index를 가진 새 `Series`.
- 원본 변경: 하지 않는다.
- 재할당: 새 feature나 target으로 쓰려면 필요하다.

### 4.5 안전성은 컬럼의 역할로 판단한다

`shift(-1)` 자체가 항상 나쁜 것은 아니다.

| 사용 위치 | 의미 | 판단 |
|---|---|---|
| `y` target | 다음 달 실제 정답 | 정상 |
| `X` feature | 미래 실제값을 입력으로 사용 | 누수 |

방향만 외우지 말고 “어디에 사용하는가”까지 확인해야 한다.

## 5. Target, baseline prediction, model prediction

### 5.1 반복했던 혼동

- `target_next_month`를 예측값으로 생각함
- baseline 예측과 모델 예측을 서로 비교해 오차를 계산함
- 예측 대상 시점과 예측값의 출처 시점을 혼동함

### 5.2 세 객체의 역할

| 객체 | 역할 | 값의 출처 |
|---|---|---|
| `target_next` | 실제 정답 | 다음 달 실제값 |
| `baseline_pred` | 기준 모델 예측 | 현재 달 값 |
| `model_pred` | ML 모델 예측 | 입력 feature와 학습된 관계 |

Persistence baseline은 다음 규칙이다.

> 다음 달도 이번 달과 같을 것이다.

```python
df["target_next"] = df["value"].shift(-1)
df["baseline_pred"] = df["value"]
```

같은 행에 있어도 둘은 다르다.

```text
baseline_pred: 지금 알고 있는 값으로 만든 예측
target_next: 나중에 실제로 관측된 정답
```

### 5.3 올바른 평가

```python
from sklearn.metrics import mean_absolute_error

baseline_mae = mean_absolute_error(
    test["target_next"],
    test["baseline_pred"],
)

model_mae = mean_absolute_error(
    test["target_next"],
    test["model_pred"],
)
```

### 5.4 잘못된 평가

```python
# 두 모델이 서로 얼마나 다른지만 측정한다.
# 실제 정답에 얼마나 가까운지는 알 수 없다.
wrong_mae = mean_absolute_error(
    test["baseline_pred"],
    test["model_pred"],
)
```

두 예측이 서로 같아도 둘 다 정답에서 멀 수 있다.

### 5.5 극단적인 누수

```python
test["baseline_pred"] = test["target_next"]
```

MAE와 RMSE가 0이 되겠지만 좋은 모델이 아니다. 실제 정답을 복사했기 때문이다.

> 낮은 오차보다 먼저 예측값의 출처를 확인한다.

# Part III. 결측치와 feature 생성 순서

## 6. 결측치는 원인을 설명해야 한다

### 6.1 구조적으로 생기는 결측

```python
df["lag1"] = df["value"].shift(1)
df["target_next"] = df["value"].shift(-1)
```

- 첫 행 `lag1=NaN`: 이전 행이 없음
- 마지막 행 `target_next=NaN`: 다음 행의 정답이 없음

이 결측은 코드 오류가 아니다. feature와 target의 시간 구조가 만든 정상적인 경계 현상이다.

### 6.2 결측 자체는 Data Leakage가 아니다

잘못된 표현:

> target 결측이 있으므로 Data Leakage다.

정확한 표현:

> target 결측 행은 정답이 없어 학습·평가에 사용할 수 없다. 결측을 미래 정보로 채우면 Data Leakage가 될 수 있다.

### 6.3 `dropna()`는 범위를 지정한다

```python
model_df = df.dropna(subset=["lag1", "rolling3", "target_next"]).copy()
```

#### `DataFrame.dropna(subset=...)`

- 목적: 지정한 필수 컬럼에 결측이 있는 행을 제외한다.
- `subset`: 검사할 컬럼명 목록.
- 반환값: 새 `DataFrame`.
- 원본 변경: 기본적으로 하지 않는다.
- 재할당: 결과를 계속 쓰려면 필요하다.

잘못된 Python 표현:

```python
df.dropna(subset=["lag1" and "target_next"])
```

`"lag1" and "target_next"`는 Python에서 `"target_next"`로 평가된다. 따라서 두 컬럼이 아니라 하나만 검사한다.

올바른 표현:

```python
df.dropna(subset=["lag1", "target_next"])
```

### 6.4 모든 결측을 무조건 제거하면 안 되는 이유

```python
df.dropna()
```

이 코드는 모델에 사용하지 않는 컬럼의 결측 때문에도 행을 제거한다. 분석 목적에 따라 필요한 컬럼을 먼저 정해야 한다.

## 7. Feature 생성 전에 행을 삭제하면 시간 의미가 바뀐다

### 7.1 잘못된 순서

```python
df = df.dropna(subset=["value"])
df["lag1"] = df["value"].shift(1)
df["rolling3"] = df["value"].rolling(3).mean()
```

중간 월의 값이 결측이었다면 그 행이 먼저 사라진다. 이후 `shift()`와 `rolling()`은 떨어진 두 달을 이웃 행으로 취급한다.

### 7.2 안전한 기본 순서

1. 날짜 변환
2. 날짜 정렬
3. 중복 날짜와 누락 월 점검
4. 원래 월별 행에서 target과 feature 생성
5. 필요한 컬럼의 공통 유효 행 선택
6. 시간순 split

```python
df["date"] = pd.to_datetime(df["date"])
df = df.sort_values("date").reset_index(drop=True)

df["target_next"] = df["value"].shift(-1)
df["lag1"] = df["value"].shift(1)
df["rolling3"] = df["value"].rolling(window=3).mean()

required = ["value", "target_next", "lag1", "rolling3"]
model_df = df.dropna(subset=required).copy()
```

### 7.3 `rolling(window=3)`의 정확한 의미

기본 `center=False`에서는 현재 행과 이전 두 행을 사용한다.

| date | value | rolling3 source |
|---|---:|---|
| 2024-03 | 30 | 1월, 2월, 3월 |

`rolling(3)`은 “최근 3개월”이라는 달력 개념이 아니라 “현재 행을 포함한 최근 3개 행”이다. 월별 연속성이 확인돼야 두 의미가 같아진다.

```python
df["rolling3"] = df["value"].rolling(
    window=3,
    min_periods=3,
    center=False,
).mean()
```

- 반환값: 먼저 `Rolling` 객체, `.mean()` 후 새 `Series`.
- 원본 변경: 하지 않는다.
- 재할당: feature로 쓰려면 필요하다.
- 위험: `center=True`는 현재 행의 양옆을 사용해 미래 행이 포함될 수 있다.

# Part IV. 현실의 시간: reference date, release date, vintage

## 8. 표의 날짜가 실제로 알게 된 날짜는 아니다

### 8.1 세 개의 시간축

경제 데이터에는 적어도 세 날짜가 있을 수 있다.

| 날짜 | 의미 |
|---|---|
| Reference period | 지표가 설명하는 경제 활동의 기준 기간 |
| Release date | 값이 실제 공개된 날짜 |
| Vintage date | 특정 시점에 저장돼 있던 데이터 버전 |

예시:

| reference month | first release | first value | revised release | revised value |
|---|---|---:|---|---:|
| 2024-03 | 2024-04-20 | 100.2 | 2024-06-20 | 100.7 |

`2024-03`은 3월에 알 수 있었다는 뜻이 아니다.

### 8.2 Prediction time에 따른 가용성

예측 시점이 2024-03-31이라면:

- 3월이라는 달력 정보: 알 수 있음
- 3월 지표의 최초 발표값: 아직 발표 전이므로 알 수 없음
- 6월에 수정된 3월 값: 당연히 알 수 없음

### 8.3 계산상 과거 feature도 누수일 수 있다

```python
df["diff1"] = df["value"].diff(1)
```

이 코드는 행 방향으로 과거만 사용한다. 하지만 현재 월 `value`가 prediction time 이후 발표된다면 현실에서는 `diff1`을 계산할 수 없다.

따라서 안전성에는 두 층이 있다.

1. 계산 방향 안전성: 미래 행을 사용하지 않는가?
2. 발표 시점 안전성: 사용한 각 값이 prediction time까지 발표됐는가?

### 8.4 Revision leakage

현재 다운로드한 최신 CSV로 과거 backtest를 하면 현재 알고 있는 수정값이 과거 행에 들어갈 수 있다.

```text
2015년 당시 모델이 본 값 ≠ 2026년에 내려받은 CSV의 2015년 값
```

이 차이를 복원하려면 historical vintage 또는 real-time dataset이 필요하다.

### 8.5 현실성 수준을 문장으로 표시하기

Vintage가 없을 때는 다음처럼 한계를 명시한다.

> 이 실험은 최신 vintage를 사용한 retrospective backtest다. 당시 최초 발표값과 release lag를 완전히 재현하지 못하므로 실제 실시간 성능으로 단정할 수 없다.

### 복습 질문

2. `shift(1)`로 만든 feature면 언제나 현실적으로 안전한가?
3. 최신 CSV의 과거 값을 그대로 사용하면 어떤 종류의 누수가 생길 수 있는가?

# Part V. Feature의 의미와 유용성

## 9. 수준, lag, rolling, 차분은 서로 다른 정보를 담는다

| Feature | 예시 수식 | 담는 정보 |
|---|---|---|
| Current level | \(x_t\) | 현재 수준 |
| Lag1 | \(x_{t-1}\) | 직전 시점 수준 |
| Rolling3 | \((x_t+x_{t-1}+x_{t-2})/3\) | 최근 평균 수준 |
| Diff1 | \(x_t-x_{t-1}\) | 최근 변화량 |

### 9.1 변화량과 변화율

반복해서 혼동했던 부분이다.

변화량:

\[
\Delta x_t=x_t-x_{t-1}
\]

변화율:

\[
r_t=\frac{x_t-x_{t-1}}{x_{t-1}}
\]

예를 들어 100에서 105가 됐다면:

- 변화량: 5포인트
- 변화율: 0.05 또는 5%

```python
df["diff1"] = df["value"].diff(1)
df["pct_change1"] = df["value"].pct_change(1)
```

`diff()`와 `pct_change()`는 둘 다 새 `Series`를 반환하고 원본을 바꾸지 않는다.

### 9.2 안전하게 만들 수 있음 ≠ 성능에 도움

Feature 판단은 세 단계를 통과해야 한다.

1. 계산 가능한가?
2. 예측 시점에 사용 가능한가?
3. Validation 성능을 개선하는가?

`month_sin`, `month_cos`는 미래 달력에서 계산 가능하지만, 최근 CLI 실험에서는 B보다 C의 MAE/RMSE가 소폭 커졌다.

따라서 정확한 결론은:

> 순환형 월 feature를 만들 수는 있지만 이번 validation에서는 추가 증분 가치가 확인되지 않았다.

틀린 결론:

> 월 feature는 항상 쓸모없다.

한 validation 구간의 결과를 모든 데이터와 기간으로 일반화할 수 없다.

---

## 10. 월 번호의 순환형 표현

### 10.1 왜 1~12를 그대로 넣는 것이 불완전한가

숫자 직선에서는:

```text
|12 - 1| = 11
```

하지만 달력에서는 12월과 1월이 이웃이다.

### 10.2 원 위 좌표

\[
month\_sin=\sin(2\pi \cdot month/12)
\]

\[
month\_cos=\cos(2\pi \cdot month/12)
\]

```python
import numpy as np

month = df["date"].dt.month
df["month_sin"] = np.sin(2 * np.pi * month / 12)
df["month_cos"] = np.cos(2 * np.pi * month / 12)
```

두 좌표를 함께 써야 원 위 위치를 구분할 수 있다.

### 10.3 언제 효과가 작을 수 있는가

- 데이터가 이미 계절조정됨
- 목표 변수에 월별 계절성이 약함
- 다른 feature가 계절 패턴을 이미 설명함
- validation 기간이 특정 국면에 치우침

Feature의 이론적 의미와 실제 증분 성능을 분리해야 한다.

# Part VI. 평가를 정확히 설계하고 읽기

## 11. Train, validation, test의 역할

| 구간 | 역할 |
|---|---|
| Train | 모델 파라미터 학습 |
| Validation | feature, 모델, hyperparameter 선택 |
| Test | 선택이 끝난 뒤 최종 일반화 성능 확인 |

### 11.1 왜 test를 반복해서 보면 안 되는가

Test 결과를 보고 feature를 추가하거나 제거하면 test 정보가 모델 선택에 반영된다. 직접 학습 데이터에 넣지 않았어도 사람의 결정 과정으로 누수가 발생한다.

```text
test 확인 → feature 변경 → test 재확인 → feature 변경
```

이 과정을 반복하면 test가 사실상 validation이 된다.

### 11.2 시간순 분할

```python
train = model_df[
    (model_df["date"] >= "2000-01-01")
    & (model_df["date"] <= "2014-12-01")
].copy()

validation = model_df[
    (model_df["date"] >= "2015-01-01")
    & (model_df["date"] <= "2019-12-01")
].copy()

locked_test = model_df[
    (model_df["date"] >= "2020-01-01")
    & (model_df["date"] <= "2024-11-01")
].copy()
```

시계열에서 무작위 분할은 미래 국면이 train에 들어가 과거를 예측하는 비현실적인 구조를 만들 수 있다.

---

## 12. 모든 후보는 같은 평가 행을 사용해야 한다

### 12.1 왜 중요한가

모델 A가 쉬운 50개월, 모델 B가 어려운 60개월에서 평가됐다면 MAE 차이가 모델 때문인지 평가 표본 때문인지 알 수 없다.

### 12.2 잘못된 방식

```python
df_A = df.dropna(subset=features_A + ["target"])
df_B = df.dropna(subset=features_B + ["target"])
```

각 모델의 행 수가 달라질 수 있다.

### 12.3 공통 행 방식

가장 큰 feature set과 target을 기준으로 공통 행을 먼저 만든다.

```python
feature_sets = {
    "A": ["lag1", "rolling3"],
    "B": ["lag1", "rolling3", "diff1"],
    "C": ["lag1", "rolling3", "diff1", "month_sin", "month_cos"],
}

required = feature_sets["C"] + ["target_next", "baseline_pred"]
common = df.dropna(subset=required).copy()
```

이후 모든 모델은 `common`에서 같은 날짜 경계를 사용한다.

### 12.4 반드시 출력할 점검값

```python
print("공통 행 수:", len(common))
print("시작:", common["date"].min())
print("끝:", common["date"].max())
print("train:", len(train))
print("validation:", len(validation))
```

결과표의 `validation_rows`는 feature 개수가 아니라 실제 평가 관측치 수다.

## 13. MAE, MSE, RMSE

실제값 \(y_i\), 예측값 \(\hat y_i\), 오차 \(e_i=y_i-\hat y_i\)라고 하자.

### 13.1 MAE

\[
MAE=\frac{1}{n}\sum |e_i|
\]

- 절대오차 평균
- 원래 target과 같은 단위
- 양수·음수 오차가 상쇄되지 않음

```python
mae = (y_true - y_pred).abs().mean()
```

잘못된 형태:

```python
# 양수와 음수가 상쇄될 수 있다.
wrong_mae = (y_true - y_pred).mean()
```

### 13.2 MSE

\[
MSE=\frac{1}{n}\sum e_i^2
\]

- 큰 오차를 더 강하게 반영
- 단위가 제곱됨

```python
mse = ((y_true - y_pred) ** 2).mean()
```

### 13.3 RMSE

\[
RMSE=\sqrt{MSE}
\]

- 큰 오차에 민감
- 제곱근을 씌워 원래 target 단위로 돌아옴

```python
import numpy as np

rmse = np.sqrt(((y_true - y_pred) ** 2).mean())
```

### 13.4 지표 이름보다 계산식을 확인한다

```python
MAE = ((y_true - y_pred) ** 2).mean()
```

변수 이름이 `MAE`여도 실제 계산은 MSE다. 변수명이 계산의 의미를 바꾸지 않는다.

### 13.5 작은 수작업 예

```text
y_true = [10, 20]
y_pred = [8, 24]
error  = [2, -4]
```

- 부호 있는 평균 오차: \((2-4)/2=-1\)
- MAE: \((2+4)/2=3\)
- MSE: \((4+16)/2=10\)
- RMSE: \(\sqrt{10}\approx3.162\)

## 14. 원본 변경, 새 객체 반환, 재할당

반복된 실수 중 기술적으로 가장 자주 나타난 패턴이다.

### 14.1 새 결과를 반환하는 대표 메서드

```python
df.sort_values("date")
df.drop_duplicates()
df.dropna(subset=["target"])
series.shift(1)
series.diff(1)
series.rolling(3).mean()
```

기본적으로 새 결과를 반환하므로 계속 사용하려면 저장해야 한다.

```python
df = df.sort_values("date")
df = df.drop_duplicates(subset=["date"])
```

### 14.2 컬럼 대입은 DataFrame을 변경한다

```python
df["lag1"] = df["value"].shift(1)
```

오른쪽 `shift()`는 새 `Series`를 반환하지만, 왼쪽 컬럼 대입은 `df`에 새 컬럼을 추가한다.

### 14.3 `fit()`은 특별하다

```python
model.fit(X_train, y_train)
```

- 입력 `X_train`, `y_train`은 변경하지 않는다.
- `model` 객체 내부에는 학습된 계수 등이 저장된다.
- 반환값도 보통 자기 자신이지만, 핵심은 기존 모델 객체가 학습 상태가 된다는 것이다.

### 14.4 `predict()` 결과는 저장해야 한다

```python
pred = model.predict(X_validation)
```

- 반환값: 행당 하나의 예측값을 담은 배열
- 입력 DataFrame 변경: 하지 않음
- 평가하려면 `pred`에 저장하거나 새 컬럼에 대입해야 함

# Part VIII. 관찰과 해석을 분리하기

## 15. 데이터가 보여주는 사실과 내가 추론한 설명

### 15.1 그래프에서 직접 말할 수 있는 것

- 특정 구간에서 값이 상승했다.
- 두 변수의 산점도에 뚜렷한 선형 패턴이 약하다.
- 특정 날짜에 결측이 있다.
- validation에서 모델 B의 MAE가 모델 A보다 낮다.

### 15.2 추가 증거 없이 말할 수 없는 것

- 정책 변화 때문에 값이 올랐다.
- 산점도 관계가 약하므로 예측에 절대 쓸모없다.
- 모델 B가 모든 미래 기간에서 더 좋다.
- feature importance가 높으므로 인과적 원인이다.

### 15.3 안전한 문장 구조

```text
관찰:
이번 validation 60행에서 B의 MAE가 A보다 낮았다.

제한:
이 결과는 해당 기간과 데이터 vintage에 대한 out-of-sample 결과다.

추가 확인:
잠긴 test와 실제 release/vintage 조건에서도 유지되는지 확인해야 한다.
```

### 15.4 “모델이 틀렸다”와 “이번 비교에서 약했다”

Baseline보다 오차가 큰 모델을 보고 “모델이 잘못됐다”고 단정하면 안 된다.

가능한 해석:

- 현재 feature가 부족함
- 선형 관계가 적합하지 않음
- validation 국면이 다름
- publication/revision 조건이 비현실적임
- 단순 persistence가 해당 지표에 매우 강함

평가 결과는 모델이 실패한 조건을 알려주는 증거이지, 하나의 원인을 자동으로 알려주지 않는다.

# Part IX. 하나로 연결한 완전한 Toy Workflow

## 16. 실행 가능한 전체 예시

아래 예시는 실제 과제와 다른 가상 월별 판매지수 데이터다.

```python
import numpy as np
import pandas as pd
from sklearn.linear_model import LinearRegression
from sklearn.metrics import mean_absolute_error, mean_squared_error

# --------------------------------------------------
# 1. 가상 원자료
# --------------------------------------------------
dates = pd.date_range("2018-01-01", periods=72, freq="MS")
t = np.arange(len(dates))

df = pd.DataFrame({
    "date": dates,
    "sales_index": (
        100
        + 0.2 * t
        + 2 * np.sin(2 * np.pi * (t % 12) / 12)
    ),
})

# --------------------------------------------------
# 2. 데이터 구조 점검
# --------------------------------------------------
df["date"] = pd.to_datetime(df["date"])
df = df.sort_values("date").reset_index(drop=True)

assert df["date"].is_monotonic_increasing
assert df["date"].duplicated().sum() == 0

expected = pd.date_range(df["date"].min(), df["date"].max(), freq="MS")
missing = expected.difference(pd.DatetimeIndex(df["date"]))
assert len(missing) == 0

# --------------------------------------------------
# 3. 원래 월별 행에서 feature와 target 생성
# --------------------------------------------------
df["target_next"] = df["sales_index"].shift(-1)
df["baseline_pred"] = df["sales_index"]
df["lag1"] = df["sales_index"].shift(1)
df["rolling3"] = df["sales_index"].rolling(3).mean()
df["diff1"] = df["sales_index"].diff(1)

month = df["date"].dt.month
df["month_sin"] = np.sin(2 * np.pi * month / 12)
df["month_cos"] = np.cos(2 * np.pi * month / 12)

# --------------------------------------------------
# 4. Feature set과 공통 행
# --------------------------------------------------
feature_sets = {
    "A": ["lag1", "rolling3"],
    "B": ["lag1", "rolling3", "diff1"],
    "C": ["lag1", "rolling3", "diff1", "month_sin", "month_cos"],
}

required = feature_sets["C"] + ["target_next", "baseline_pred"]
model_df = df.dropna(subset=required).copy()

# --------------------------------------------------
# 5. 시간순 train/validation/test
# --------------------------------------------------
train = model_df[model_df["date"] <= "2021-12-01"].copy()

validation = model_df[
    (model_df["date"] >= "2022-01-01")
    & (model_df["date"] <= "2022-12-01")
].copy()

locked_test = model_df[
    model_df["date"] >= "2023-01-01"
].copy()

print("train rows:", len(train))
print("validation rows:", len(validation))
print("locked test rows:", len(locked_test))

# --------------------------------------------------
# 6. Baseline 평가
# --------------------------------------------------
results = []

baseline_mae = mean_absolute_error(
    validation["target_next"],
    validation["baseline_pred"],
)
baseline_rmse = mean_squared_error(
    validation["target_next"],
    validation["baseline_pred"],
) ** 0.5

results.append({
    "model": "baseline",
    "feature_count": 1,
    "validation_rows": len(validation),
    "MAE": baseline_mae,
    "RMSE": baseline_rmse,
})

# --------------------------------------------------
# 7. A/B/C 모델을 같은 행에서 비교
# --------------------------------------------------
for name, features in feature_sets.items():
    model = LinearRegression()
    model.fit(train[features], train["target_next"])

    pred = model.predict(validation[features])

    results.append({
        "model": name,
        "feature_count": len(features),
        "validation_rows": len(validation),
        "MAE": mean_absolute_error(validation["target_next"], pred),
        "RMSE": mean_squared_error(validation["target_next"], pred) ** 0.5,
    })

result_df = pd.DataFrame(results)
print(result_df)

# locked_test는 feature/model 선택이 끝날 때까지 평가하지 않는다.
```

# Part XI. 오답 교정 훈련

## 20. 다음 문장을 교정해 보기

### 문장 1

> 날짜가 오름차순이므로 모든 월이 연속적으로 존재한다.

교정 핵심:

- 정렬과 연속성은 별도다.
- 예상 날짜와 실제 날짜의 차이를 확인해야 한다.

### 문장 2

> `shift(-1)`은 미래를 사용하므로 언제나 Data Leakage다.

교정 핵심:

- Target 생성에는 정상이다.
- Feature X에 넣을 때 누수다.

### 문장 3

> 마지막 target이 결측이므로 Data Leakage다.

교정 핵심:

- 다음 달 정답이 데이터에 없어서 생긴 구조적 결측이다.
- 미래 정보로 채우면 누수 위험이 생긴다.

### 문장 4

> 모델 예측과 baseline 예측의 MAE를 계산하면 더 좋은 모델을 알 수 있다.

교정 핵심:

- 두 예측 사이의 차이만 알 수 있다.
- 각각을 같은 실제 정답과 비교해야 한다.

### 문장 5

> Lag feature는 과거 값이므로 항상 예측 시점에 사용할 수 있다.

교정 핵심:

- Reference period가 과거여도 release date가 prediction time 이후일 수 있다.
- 현재 CSV의 과거 값이 사후 수정값일 수 있다.

### 문장 6

> `diff1`은 전월 대비 변화율이다.

교정 핵심:

- `diff1`은 변화량이다.
- 변화율에는 이전 값으로 나누는 과정이 필요하다.

### 문장 7

> 월 feature를 추가했는데 validation 오차가 커졌으므로 월 정보는 모든 예측에서 쓸모없다.

교정 핵심:

- 이번 데이터, feature set, validation 기간에서 증분 가치가 없었다.
- 다른 데이터와 기간으로 일반화할 수 없다.

### 문장 8

> 현재 CSV에서 2015년 값이 보이므로 2015년 당시 모델도 그 값을 알 수 있었다.

교정 핵심:

- Release date와 historical vintage를 확인해야 한다.
